# V. 边缘辅助的可追责双撤销访问控制协议

## A. 设计动机

本文关注的是 IoV 业务数据的隐私保护分发，例如事故报告、诊断日志、路侧感知摘要以及面向任务的协同记录。
与高频安全广播不同，这类消息并不天然是公开的。它们同时要求发送者认证、接收者授权、撤销支持以及事后追责。
现有车联网认证方案和基于 ABE 的访问控制方案通常只能分别解决其中一部分问题，而且往往是彼此割裂地处理。
为弥合这一缺口，我们设计了一种边缘辅助协议，将“可追责匿名认证”和“双撤销 CP-ABE”耦合起来。

该协议遵循三个设计原则。第一，公众验证者应当能够验证消息是否合法，但不能直接获知发送者真实身份。
第二，业务数据只有在接收方属性满足当前访问策略且未被撤销的情况下才能被解密。
第三，只要能够安全地下沉到 RSU 或边缘节点，OBU 就不应承担高配对开销。

## B. 系统实体与记号

系统包含可信权威机构 TA、追踪权威机构 TrA、属性权威与撤销管理器 AA/RM、作为边缘代理的 RSU、
云端或服务平台 CSP、带有防篡改设备 TPD 的车辆 OBU，以及被授权的数据接收方。TA 与 TrA 共同支持条件隐私与可验证追责；
AA/RM 负责属性密钥与撤销更新；RSU 负责批验证与外包解密。

| 记号 | 含义 |
| --- | --- |
| `ID_i` | 车辆 `V_i` 的真实身份 |
| `PID_i^tau` | 时间槽 `tau` 下的伪名认证句柄 |
| `A_i` | 车辆或请求方 `i` 的属性集合 |
| `T` | 访问树或等价的 LSSS 策略 |
| `e_u` | 用户撤销纪元 |
| `ver(a)` | 属性 `a` 的当前版本号 |
| `KU_t` | 时间 `t` 面向未撤销用户的纪元更新材料 |
| `AU_{a,t}` | 时间 `t` 面向属性 `a` 的版本更新令牌 |
| `TK_i` | 用于外包解密的转换密钥 |
| `sigma` | 匿名且可追责的认证标签 / 签名 |
| `ctx` | 绑定到密文中的认证 transcript 哈希 |

## C. 系统初始化与注册

在系统初始化阶段，TA 建立匿名认证子系统，并将开启材料在 TA 与 TrA 之间分发。AA/RM 初始化
CP-ABE 公共参数、用户撤销树以及属性版本表。每辆车完成一次注册后，获得以下材料：

1. 与 TPD 绑定的长期身份记录；
2. 支持 `Sign`、`Verify`、`Open` 和 `Judge` 的匿名认证凭据；
3. 与其初始属性集合对应的属性私钥；
4. 与二叉撤销树路径相关的撤销秘密；
5. 面向 RSU 辅助解密的转换密钥导出句柄。

TA 与 AA/RM 在逻辑上应当分离，以避免单一机构同时拥有“追踪用户”和“解密数据”的双重能力。

## D. 匿名认证与 transcript 绑定

当车辆 `V_s` 向 RSU 或服务平台发送业务数据消息时，首先需要对消息头进行匿名认证。
设公开头部为 `m_hdr`，其中包含业务类型、粗粒度区域标签以及当前时间槽 `tau`。发送方生成：

`sigma <- Sign(cred_s, m_hdr, tau)`。

任意验证者都可以执行 `Verify(pp_auth, m_hdr, tau, sigma)`，确认消息来自某个合法但身份隐藏的发送者。
为了把认证与授权绑定起来，发送方进一步计算：

`ctx = H(m_hdr || tau || sigma || sid || e_u)`，

其中 `sid` 是一次数据分发会话标识。随后，`ctx` 会作为不可篡改的绑定标签被嵌入 ABE 密文中。
这样设计的作用是防止攻击者拿旧的匿名认证 transcript 去包装新的受保护载荷，或者把合法签名与不相关的访问策略拼接起来。

如果发生争议，TA 与 TrA 将联合执行 `Open` 恢复对应的长期身份，并出具追踪声明。公开的 `Judge` 过程可验证恢复出的身份是否与
`(m_hdr, tau, sigma)` 一致，并验证追踪方没有伪造结论。这样就实现了“对公众匿名、对授权机构可验证追责”。

## E. 基于策略的业务数据分发

在完成头部认证后，发送方采样对称会话密钥 `K`，并利用带认证的对称加密算法保护载荷 `M`：

`C_dem = AEAD.Enc(K, M, aad = m_hdr || tau || sigma)`。

随后，发送方定义一个“部分隐藏”的访问策略。公开部分只暴露粗粒度业务语义，而敏感策略叶子尽量隐藏。
一个具有代表性的策略为：

`T = ((Role AND Region AND Task) OR Emergency) AND Freshness`。

发送方用 CP-ABE 在策略 `T` 下加密 `(K, ctx, meta)`，其中 `meta` 包含当前用户撤销纪元 `e_u`，
以及参与策略的属性当前版本 `ver(a)`。最终的数据分发对象记为：

`C = {m_hdr, tau, sigma, C_dem, CT_abe}`。

这样，公开验证仍然是轻量的，而真正的业务数据访问权限则由满足当前策略的属性集合决定。

## F. 边缘辅助外包解密

为了降低 OBU 时延，每个合法接收方都会导出一个轻量转换密钥 `TK_i`，并向所服务的 RSU 提交最小必要的验证材料。
RSU 接收到 `C` 后，先本地或批量验证 `sigma`。若消息合法，则执行：

`CT'_i <- Transform(TK_i, CT_abe)`。

转换后的密文不会向 RSU 泄露有效明文信息，但会把绝大部分配对密集型工作从 OBU 转移出去。
接收方随后只需执行一个轻量终解密步骤：

`(K, ctx') <- FinalDec(sk_i, CT'_i)`。

只有当 `ctx' = H(m_hdr || tau || sigma || sid || e_u)` 时，OBU 才接受输出结果。这个一致性检查非常关键。
即使恶意中继重放旧密文组件，或者把一个合法认证 transcript 和新的 ABE 外壳混合，最终也会因 transcript 绑定检查失败而无法成功解密。

## G. 双撤销机制

### 1) 用户撤销

用户撤销采用基于时间纪元的二叉树机制。每个注册用户被映射到一片叶节点，并持有从根到叶路径上的节点秘密。
在纪元 `t`，AA/RM 为所有未撤销用户的覆盖节点集合发布 `KU_t`。合法接收方可将自身路径秘密与 `KU_t` 结合，恢复当前可用的解密能力。
被撤销用户即使保留历史属性材料，也无法重建当前纪元所需密钥。

该机制的优点是更新大小随覆盖节点集合增长，而不是随用户总数线性增长。因此，它天然适合在实验中衡量：
随着撤销用户数变化，更新大小和更新应用时间如何变化。

### 2) 属性撤销

每个属性 `a` 都维护一个版本号 `ver(a)`。当某个属性从用户身上被撤销，或者该属性需要按新条件重新签发时，AA/RM 会增加 `ver(a)`，
并向所有仍合法持有该属性的用户发放新的更新令牌 `AU_{a,t}`。发送方始终针对当前版本信息进行加密。已有密文则可以由 CSP 或 RSU 利用代理更新材料进行惰性刷新，
从而避免由原始发送者立即执行完全重加密。

该设计把“用户身份是否仍有效”和“某项权限是否仍保留”区分开来。也就是说，某辆车仍可能是合法网络参与者，
但它可能已经失去 `Maintenance`、`Law-Enforcement` 或 `Fleet-Coordinator` 等某一项角色属性。
在 IoV 场景中，这种区分非常重要，因为服务权限的变化频率通常高于注册身份状态的变化频率。

## H. 可验证追责流程

可验证追责只在按需触发时执行。当系统收到投诉，或识别出恶意数据分发事件后，TA 与 TrA 联合运行 `Open(m_hdr, tau, sigma)`，
恢复发送方真实身份 `ID_s`。之后输出追责记录：

`TR = {m_hdr, tau, sigma, ID_s, pi_open, sig_TA, sig_TrA}`。

其中，`pi_open` 是证明开启过程与匿名认证系统一致的验证对象，`sig_TA` 与 `sig_TrA` 用于证明追责结果确实经过联合授权。
任何审计者都可以执行 `Judge(TR)` 来验证该追责结果是否有效、是否不可伪造。这样一来，条件隐私不再只是“相信权威机构不会滥用权力”，
而是升级为“可公开验证的追责机制”。

## I. 安全性承接段

该协议的安全目标可以直接从其组合结构导出：匿名认证不可伪造、发送者对公众匿名、联合开启下可追责、业务数据仅能在满足访问策略时被恢复、
具有抗串谋能力、并且外包解密过程正确且不破坏机密性。在论文中，可以将这些目标形式化为：
把可追责匿名认证层的安全性，与部分隐藏 CP-ABE 在 transcript 绑定下的 IND-CPA 或 IND-sAtt-CPA 风格安全性结合起来进行证明。

# VI. 实验方法与评估设计

## A. 评估目标

实验部分应回答一个统一的系统问题，而不是拆成若干彼此割裂的密码学问题：
在保护 IoV 业务数据的前提下，本文方案是否能够在保持可追责隐私保护的同时，降低 OBU 侧开销并维持较低的端到端分发时延？
为了在 GLOBECOM 的篇幅内清晰回答这一问题，实验应由“密码学微基准 + 网络级仿真”组成。

## B. 原型实现栈

建议采用两层原型结构。

1. **密码原型层。**
   使用统一框架实现协议原语。`Charm` 或 `OpenABE` 可用于验证策略处理与功能正确性；
   `JPBC` 或 `PBC/C` 则用于最终计时，因为实验需要精确测量配对和指数运算代价。

2. **网络仿真层。**
   使用 `Veins + OMNeT++ + SUMO` 建模数据分发网络。把密码原型层测得的时延，
   作为发送方 OBU、接收方 OBU 和 RSU 的应用层处理时延注入仿真器。

这种拆分方法可以保证实验更可复现，也能避免一种常见问题：没有测量底层密码学代价，却直接声称得到了端到端网络性能。

## C. baseline 选择

baseline 应当与 related work 矩阵保持一致，并且每一个 baseline 都要对应一个明确的对比问题。

| Baseline | 评估角色 | 复用指标 |
| --- | --- | --- |
| BCPPA [1] | 仅认证的隐私保护 baseline | 签名 / 验证时延、通信开销 |
| SE-CPPA [2] | 轻量匿名认证 baseline | 批验证收益、认证时延 |
| Dynamic FGAC [3] | 访问控制与用户撤销 baseline | 授权时延、更新开销 |
| Lightweight direct-AR CP-ABE [4] | 属性撤销与 RSU 卸载 baseline | OBU 终解密时延、RSU 转换开销 |
| Flexible Revocation ABE [5] | 基础用户撤销机制 baseline | 更新大小、纪元重密钥时延 |
| TRE-DSP [6] | 最接近本文的 IoV 数据共享 baseline | 追责时延、密文开销、撤销感知分发 |

所有 baseline 最好都在统一安全级别下进行重实现或标准化。论文中可以引用其原始实验结果，但不能把它们直接作为公平对比的主依据。

## D. 核心指标

建议把实验指标收敛到以下集合。

| 类别 | 指标 | 含义 |
| --- | --- | --- |
| OBU 侧开销 | 签名时延 | 生成匿名认证标签所需时间 |
| OBU 侧开销 | 终解密时延 | RSU 转换后客户端剩余的解密时间 |
| RSU 侧开销 | 转换开销 | 边缘侧外包解密代价 |
| 追责能力 | 追责时延 | 从发起追责请求到 `Judge` 验证完成的时间 |
| 撤销开销 | 用户更新大小 | 在不同撤销用户数下 `KU_t` 的大小 |
| 撤销开销 | 属性更新开销 | `AU_{a,t}` 的大小与应用时延 |
| 网络性能 | 端到端投递时延 | 从发送端生成数据到授权接收端恢复明文的时间 |
| 网络性能 | PDR / 吞吐量 | 负载下的投递有效性 |
| 网络性能 | 通信开销 | 认证与 ABE 元数据引入的额外字节数 |

这些指标能够直接支撑全文主论点：OBU 负担更低、撤销更高效、并且具备可部署的隐私保护数据分发能力。

## E. 参数设置

建议实验扫描如下参数范围。

| 参数 | 取值 |
| --- | --- |
| 用户属性数 | 5, 10, 20, 30, 50 |
| 策略行数 / 叶节点数 | 5, 10, 20, 40 |
| 每纪元撤销用户数 | 1, 10, 50, 100 |
| 每次更新撤销属性数 | 1, 5, 10 |
| 车辆总数 | 50, 100, 200, 500 |
| RSU 数量 | 4, 8, 16 |
| 业务数据载荷大小 | 200, 500, 1000 bytes |

所有方案应首先标准化到大致 128-bit 安全级别。如果某些旧 baseline 依赖过时配对参数，
则可以额外给出一组“原始参数复现曲线”，用于解释与标准化结果之间的差异来源。

## F. 微基准实验流程

对于每组参数，建议测量以下项目：

1. 发送方签名时间；
2. 发送方 ABE 加密时间；
3. RSU 侧验证与转换时间；
4. 接收方终解密时间；
5. 用户撤销更新生成与应用时间；
6. 属性撤销更新生成与惰性刷新时间；
7. 争议场景下的开启与判定时间。

每个数据点至少运行 30 次，并报告平均值与 95% 置信区间。所有方案都应在相同机器、相同曲线族、
相同编译器或 JVM 配置以及相同库版本下测试。

## G. 网络级仿真

网络级实验应聚焦于“受保护的业务数据分发”，而不是周期性安全广播。一个较实用的设置是：
在 SUMO 中构造混合城市路网场景，车辆数从 50 到 500，RSU 数量为 4 到 16。车辆以中等速率产生业务数据包，
且只有满足当前策略的接收方能够恢复载荷。仿真器在应用层注入密码学处理时延，然后报告：

- 端到端投递时延；
- 分组投递率；
- 授权接收者的有效吞吐量；
- 追责事件或撤销刷新事件带来的额外时延。

这种设计比把 CP-ABE 直接套到 10 Hz 安全广播上更合理，也能有效避免审稿人质疑“你为什么把重型密码学强行用于硬实时广播”。

## H. 消融实验

五组消融就足够，而且与论文贡献点一一对应。

| 变体 | 去掉的能力 | 预期结果 |
| --- | --- | --- |
| 无用户撤销 | 移除基于纪元的二叉树更新 | 更新更轻，但历史用户仍可继续解密 |
| 无属性撤销 | 移除版本化属性更新 | 管理开销更低，但无法细粒度移除权限 |
| 无外包解密 | 强制 OBU 完成全部 ABE 解密 | OBU 时延更高，可扩展性更差 |
| 无可验证追责 | 用不透明追踪替代 `Open/Judge` | 控制开销略低，但追责可信度下降 |
| 无 transcript 绑定 | 移除 `ctx` 一致性检查 | 原始开销接近，但更容易遭受混合重放攻击 |

这些消融有助于让审稿人看到：每个设计部件都对应一个明确的系统收益，而不是堆砌功能。

## J. 论文中适合声称的结论

如果没有得到更强的数据支撑，正文建议只做以下四类结论：

1. 本文方案在保持匿名认证的同时，实现了可验证追责；
2. 双撤销机制可以同时移除被撤销用户和被撤销权限；
3. RSU 辅助外包解密把主要配对开销从资源受限的 OBU 转移出去；
4. 在现实的 IoV 业务数据工作负载下，端到端时延仍然适合低时延保护性分发。

这种表述足够有力度，同时也更容易在 GLOBECOM 审稿中站得住。
